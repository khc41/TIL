### 5장. 복제
- 데이터 복제가 필요한 이유
  - 지리적으로 가깝게 유지해 지연을 줄임
  - 장애가 발생해도 지속적으로 동작할 수 있게 가용성 높임
  - 읽기 처리량을 늘림
- 복제에서 모든 어려움은 복제된 데이터의 변경 처리에 있다.
- 복제에는 동기식 복제 / 비동기식 복제, 잘못된 복제본 처리 등의 트레이드 오프가 있고 선택이 중요하다.

#### 리더와 팔로워
- 데이터베이스의 복사본을 저장하는 각 노드를 복제 서버(replica)라고 한다.
- 모든 쓰기는 모든 복제 서버에서 처리되어야 한다.
  - 이를 해결하기 위해 일반적으로 리더 기반 복제가 있다.
    - 복제 서버중 하나를 리더로 지정하고 쓰기를 요청할 때는 리더에게 요청을 보낸다.
    - 다른 복제 서버는 팔로워인데 리더가 기록할 때마다 데이터 변경을 복제 로그나 변경 스트림의 일부로 팔로워에게 전송한다.

##### 동기식 대 비동기식 복제
- 복제 시스템은 동기식인지 비동기식인지가 중요하다.
- 팔로워에게 변경 내용을 적용할 때 팔로워가 수분 이상 리더와 떨어질 수 있고 장애를 복구 중이거나 시스템이 최대 가용량 부근에서 동작하거나 노드 간 네트워크 문제가 있을 수 있기 때문에 얼마나 오래 걸릴지 보장할 수 없다.
- 동기식 복제의 장점은 팔로워가 리더가 일관성 있게 최신 데이터 복사본을 가지는 것을 보장한다.
- 단점은 동기 팔로워가 응답하지 않는다면 쓰기가 처리될 수 없다.
- 현실적으로 동기식 복제를 사용하려면 팔로워 하나는 동기식으로 하고 그 밖에는 비동기식으로 한다.
  - 적어도 두 노드에 최신 복사본이 있는 것을 보장하는데 이런 설정을 반동기식이라 한다.
- 리더 기반 복제는 완전 비동기식으로 구성하는데 리더가 잘못되고 복구할 수 없으면 팔로워에 복제되지 않은 모든 쓰기는 유실되지만 모든 팔로워가 잘못되더라도 리더가 쓰기 처리를 계속할 수 있다.

##### 새로운 팔로워 설정
- 새로운 팔로워 추가 시 데이터 복제 방법
  1. 리더의 데이터베이스 스냅숏을 일정 시점에 가져온다.
  2. 스냅숏을 새로운 팔로워 노드에 복사한다.
  3. 리더에 연결해 스냅숏 이후 발생한 모든 데이터 변경을 요청한다. 
  4. 팔로워가 스냅숏 이후 데이터 변경의 미처리분을 모두 처리했을 때 따라잡았다고 말한다.

##### 노드 중단 처리
- 중단시간 없이 개별 노드를 재부팅할 수 있다는 점은 운영과 유지보수에 큰 장점이다.
- 개별 노드 장애에도 전체 시스템이 동작하게끔 유지하고 노드 중단 영향을 최소화해야 한다.
- **팔로워 장애: 따라잡기 복구**
  - 각 팔로워는 리더로부터 수신한 데이터 변경 로그를 로컬 디스크에 보관한다.
  - 보관된 로그에서 결함이 발생하기 전에 처리한 마지막 트랜잭션을 알아내 연결이 끊어진 동안 발생한 데이터 변경을 모두 요청한다.
- **리더 장애: 장애 복구**
  - 장애복구 과정은 팔로워 중 하나를 새로운 리더로 승격하고 클라이언트는 새로운 리더로 쓰기를 전송하기 위해 재설정이 필요하며 다른 팔로워는 새로운 리더로부터 데이터 변경을 소비하기 시작해야한다.
    1. 리더가 장애인지 판단한다.
       - 대부분 시스템은 단순한 타임아웃을 사용한다.
    2. 새로운 리더를 선택한다.
       - 선출 과정을 통해 이뤄지거나 이전에 선출된 제어 노드에 의해 새로운 리더가 임명된다.
    3. 새로운 리더 사용을 위해 시스템을 재설정한다.
       - 새로운 쓰기 요청을 새로운 리더에게 보내고 시스템은 이전 리더가 팔로워가 되고 새로운 리더를 인식할 수 있게 해야한다.
  - 발생할 수 있는 문제
    - 비동기식 복제에서 새로운 리더는 이전 리더가 실패하기 전에 이전 리더의 쓰기 일부를 수신하지 못할 수 있다. 일반적인 해결책은 이전 리더의 복제되지 않은 쓰기를 단순히 폐기하는 방법이다.
    - 쓰기를 폐기하는 방법은 데이터베이스 외부의 다른 저장소 시스템이 데이터베이스 내용에 맞춰 조정돼야 한다면 특히 위험하다.
    - 두 노드가 자신이 리더라고 믿을 수 있는데 이를 스플릿 브레인이라 한다. 이는 데이터가 유실되거나 오염되며 일부 시스템은 안전장치로 한 노드를 종료하는 메커니즘이 있는데 두 개의 노드 모두 종료될 수 있다.
    - 일시적인 부하 급증으로 노드의 응답 시간이 타임아웃보다 커지거나 네트워크 고장으로 패킷이 지연되는 경우 등으로 인해 타임아웃을 설정하기 어렵다.
  - 쉬운 해결책이 없어 자동 장애 복구를 지원하더라도 수동으로 수행하는 것을 선호한다.

##### 복제 로그 구현
- **구문 기반 복제**
  - 리더는 모든 쓰기 요청(구문)을 기록하고 쓰기를 실행한 다음 구문 로그를 팔로워에게 전송한다. 
  - 복제가 깨질 수 있는 사례
    - NOW() 나 RAND() 같은 비결정적 함수를 호출하는 모든 구문은 각 복제 서버마다 다른 값을 생성할 가능성이 있다.
    - 자동증가 칼럼을 사용하는 구문이나 데이터베이스에 있는 데이터에 의존한다면 구문은 각 복제 서버에서 정확히 같은 순서로 실행돼야 한다.
    - 부수 효과를 가진 구문(트리거, 스토어드 프로시저, 사용자 정의 함수)은 부수 효과가 완벽하게 결정적이지 않으면 각 복제 서버에서 다른 부수 효과가 발생할 수 있다.
  - 구문을 기록할 때 모든 비결정적 함수 호출을 고정 값을 반환하게끔 대체할 수 있지만 여러 에지케이스 때문에 다른 복제 방법을 선호한다.
- **쓰기 전 로그 배송**
  - 모든 쓰기는 추가 전용 바이트 열인 로그에 기록된다.
  - 리더는 디스크에 로그를 기록하는 일 외에도 팔로워에게 네트워크로 로그를 전송한다.
  - 가장 큰 단점은 로그가 제일 저수준의 데이터를 기술한다는 점이다.
    - 복제가 저장소 엔진과 밀접하게 엮여서 리더와 팔로워의 데이터베이스 소프트웨어 버전을 다르게 실행할 수 없다.
    - 팔로워가 리더보다 새로운 소프트웨어 버전을 사용하게끔 복제 프로토콜이 허용한다면 팔로워를 먼저 업레이드함으로써 중단시간 없이 데이터베이스 소프트웨어 업그레이드 수행이 가능하다.
      - 업그레이드된 노드 중 하나를 새로운 리더로 선정하기 위해 장애 복구를 수행할 수 있다.
- **논리적(로우 기반) 로그 복제**
  - 복제와 저장소 엔진 내부를 분리하기 위한 대안은 복제와 저장소 엔진을 위해 다른 로그형식을 사용하는 것이다. 이 같은 종류의 복제 로그를 논리적 로그라고 부른다.
  - 관계형 데이터베이스용 논리적 로그는 로우 단위로 데이터베이스 테이블에 쓰기를 기술한 레코드 열이다.
    - 삽입된 로우의 로그는 모든 칼럼의 새로운 값을 포함한다.
    - 삭제된 로우의 로그는 로우를 고유하게 식별하는 데 필요한 정보를 포함한다. (기본키)
    - 갱신된 로우의 로그는 로우를 고유하게 식별하는 데 필요한 정보와 모든 칼럼의 새로운 값을 포함한다.
  - 여러 로우를 수정하는 트랜잭션은 여러 로그 레코드를 생성한 다음 트랜잭션이 커밋됐음을 레코드에 표시한다.
  - 하위 호환성을 더 쉽게 유지할 수 있고 리더와 팔로워에서 다른 버전의 데이터베이스 소프트웨어나 저장소 엔진을 실행할 수 있다.
  - 외부 애플리케이션이 파싱하기 쉬워 오프라인 분석이나 외부시스템 전송에 유용하고 이 기술을 변경 데이터 캡처라고 부른다.
- **트리거 기반 복제**
  - 데이터의 서브셋만 복제하거나 데이터베이스를 다른 종류의 데이터베이스로 복제해야 하거나 충돌 해소로직이 필요하다면 복제를 애플리케이션 층으로 옮겨야 한다.
  - 트리거나 스토어드 프로시저를 사용한다.
  - 다른 복제 방식보다 많은 오버헤드가 있고 버그나 제한 사항이 더 많이 발생하지만 유연성 때문에 매우 유용하다.

#### 복제 지연 문제
- 복제는 내결함성뿐만 아니라 확장성, 지연 시간에 필요하다.
- 대부분이 읽기 요청이고 쓰기가 아주 작은 비율이라면 많은 팔로워를 만들어 읽기 요청을 분산하는 방법이 효과적이다.
  - 이런 읽기 확장은 비동기식 복제에서만 동작한다.
- 비동기 팔로워에서 팔로워가 뒤쳐진다면 지난 정보를 볼 수 있지만, 잠시 기다리면 리더와 일치하게 되는데 이를 최종적 일관성이라 한다.

##### 자신이 쓴 내용 읽기
- 사용자가 임의 데이터를 제출하고 해당 사용자에게 제출한 데이터를 볼 수 있게 하는 경우 쓰기를 수행한 직후 데이터를 본다면 새로운 데이터는 복제 서버에 반영되지 않을 수 있는데, 이런 상황에서는 쓰기 후 읽기 일관성이 필요하다.
  - 사용자가 수정한 내용을 읽을 때는 리더에서 읽고 그 밖에서는 팔로워에서 읽는다.
  - 대부분의 내용을 사용자가 편집할 가능성이 있다면 이런 방식은 효율적이지 않기 때문에 마지막 갱신 후 1분 동안은리더에서 모든 읽기를 수행하는 등의 다른 기준을 사용해야 한다.
  - 클라이언트의 타임스탬프로 복제 서버가 해당 타임스탬프까지 갱신을 반영하게 한다. 
  - 복제서버가 여러 데이터센터에 분산됐다면 리더가 제공해야 하는 모든 요청은 리더가 포함된 데이터센터로 라우팅돼야 한다.
- 동일한 사용자가 여러 디바이스로 접근할 때 디바이스 간 쓰기 후 읽기 일관성이 제공돼야 한다.
  - 사용자의 마지막 갱신 타임스탬프를 기억하는 방식은 디바이스간 갱신을 알지 못하므로 중앙집중식으로 관리해야 한다.
  - 복제 서버가 여러 데이터센터 간에 분산돼 있다면 다른 디바이스 연결이 동일한 데이터센터로 라우팅된다는 보장이 없으므로 사용자 디바이스의 요청을 동일한 데이터센터로 라우팅해야 한다.

##### 단조 읽기
- 사용자가 시간이 거꾸로 흐르는 현상을 목격할 수 있다.
  - 각기 다른 복제 서버에서 여러 읽기를 수행할 때 발생할 수 있다.
- 단조 읽기는 이런 현상이 발생하지 않음을 보장하는데, 강한 일관성보다는 덜하고 최종적 일관성보다는 강한 보장이다.
- 각 사용자의 읽기가 항상 동일한 복제 서버에서 수행되게끔 하는 것으로 달성 가능하다.
  - 임의 선택보다는 사용자 ID의 해시를 기반으로 복제 서버를 선택한다.

##### 일관된 순서로 읽기
- 복제 지연이 있는 경우 인과성의 위반 우려가 있다.
- 일관된 순서로 읽기는 이런 이상 현상을 방지하는데 일련의 쓰기가 특정 순서로 발생한다면 이 쓰기를 읽는 모든 사용자는 같은 순서로 쓰여진 내용을 보게 됨을 보장한다.
- 파티셔닝된(샤딩된) 데이터베이스에서 발생하는 문제인데, 많은 분산데이터베이스에서 서로 다른 파티션은 독립적으로 동작하므로 쓰기의 전역 순서는 없다.
- 서로 인과성이 있는 쓰기가 동일한 파티션에 기록되게끔 하는 방법이 있는데 일부 애플리케이션에선 효율적이지 않다.

##### 복제 지연을 위한 해결책
- 최종적 일관성 시스템으로 작업할 때 복제 지연이 몇 분이나 몇 시간으로 증가한다면 애플리케이션이 어떻게 동작할지 생각해 볼 가치가 있다.
  - 사용자에게 좋지 않은 경험이라면 쓰기 후 읽기와 같은 강한 보장을 제공하게끔 시스템을 설계해야 한다.
- 애플리케이션 코드에서 이런 문제를 다루기에는 너무 복잡해서 잘못되기 쉽고 걱정하지 않고 데이터베이스를 신뢰하기 위해 트랜잭션이 있다.
- 분산 데이터베이스로 전환하는 과정에서 많은 시스템이 트랜잭션을 포기했고 어쩔수 없이 최종적 일관성을 사용해야 한다는 주장이 있다.

#### 다중 리더 복제
- 리더 기반 복제는 리더에 연결할 수 없다면 데이터베이스에 쓰기를 할 수 없다.
- 쓰기를 허용하는 노드를 하나 이상 두는 것으로 자연스럽게 확장되는데 이 방식을 다중 리더 설정이라고 부른다.

##### 다중 리더 복제의 사용 사례
- 단일 데이터센터 내에 다중 리더 설정을 사용하는 것은 복잡도에 비해 이점이 크지 않기 때문에 적절하지 않다.
- **다중 데이터센터 운영**
  - 다중 리더 설정에서는 각 데이터센터마다 리더가 있을 수 있고, 각 데이터센터의 리더가 다른 데이터센터의 리더에게 변경 사항을 복제한다.
    - 성능
      - 단일 리더 설정에서 모든 쓰기는 인터넷을 통해 리더가 있는 데이터센터로 이동해야하기 때문에 쓰기 지연 시간을 상당히 늘리는 원인이 된다. 
      - 다중 리더 설정에서는 모든 쓰기는 로컬 데이터센터에서 처리한 다음 비동기 방식으로 다른 데이터센터에 복제하기 때문에 데이터센터 간 네트워크 지연은 사용자에게 숨겨진다.
    - 데이터센터 중단 내성
      - 단일 리더 설정에서는 리더가 있는 데이터센터가 고장 나면 다른 데이터센터에서 한 팔로워를 리더로 승진 시키지만, 다중 리더 설정에서는 각 데이터센터는 다른 데이터센터와 독립적으로 동작하고 고장 난 데이터센터가 온라인으로 돌아왔을 때 복제를 따라잡는다.
    - 네트워크 문제 내성
      - 단일 리더 설정에서는 데이터센터 내 연결의 쓰기는 동기식이기 때문에 데이터센터 내 연결 문제에 매우 민감한데, 비동기 복제를 사용하는 다중 리더 설정에서는 일시적인 네트워크 중단에도 쓰기 처리는 진행되기 때문에 네트워크 문제에 보다 잘 견딘다.
  - 다중 리더 복제는 동일한 데이터를 다른 두 개의 데이터센터에서 동시에 변경할 수 있다는 단점이 있어서 쓰기 충돌을 반드시 해소해야 한다.
  - 자동 증가 키, 트리거, 무결성 제약은 문제가 될 소지가 많으므로 다중 리더 복제는 피해야 하는 위험한 영역으로 간주되곤 한다.
- **오프라인 작업을 하는 클라이언트**
  - 인터넷 연결이 끊어진 동안 애플리케이션이 계속 동작해야 하는 경우 적합하다.
  - 모든 디바이스에는 리더처럼 동작하는 로컬 데이터베이스가 있고 모든 디바이스 상에서 복제 서버 간 다중 리더 복제를 비동기 방식으로 수행하는 프로세스가 있다. 
- **협업 편집**
  - 동시에 여러 사람이 문서를 편집할 수 있는 애플리케이션을 실시간 협업 편집 애플리케이션이라 한다.
  - 협업 편집을 데이터베이스 복제 문제로 생각하지 않지만 협업 편집은 오프라인 편집 사용 사례와 공통점이 많다.
    - 로컬 복제 서버에 적용하고 나서 동일한 문서를 편집하는 다른 사용자와 서버에 비동기 방식으로 복제한다.
  - 편집 충돌이 없음을 보장하려면 편집하기 전에 문서의 잠금을 얻어야 한다. 이는 리더에서 트랜잭션을 사용하는 단일 리더 복제와 동일하다.
  - 더 빠른 협업을 위해 변경 단위를 매우 작게해서 잠금을 피할 수 있는데 충돌 해소를 필요한 경우를 포함해 다중 리더 복제에서 발생하는 모든 문제를 야기한다.

##### 쓰기 충돌 다루기
- 다중 리더 복제의 제일 큰 문제는 쓰기 충돌이 발생하는 것이고 충돌 해소가 필요하다.
- **동기 대 비동기 충돌 감지**
  - 충돌 감지는 동기식으로 만들 수 있다.
    - 쓰기가 성공한 사실을 사용자에게 말하기 전에 모든 복제 서버가 쓰기를 복제하기를 기다리는 것이 있는데 다중 리더 복제의 주요 장점을 잃는다.
- **충돌 회피**
  - 특정 레코드의 모든 쓰기가 동일한 리더를 거치도록 애플리케이션이 보장한다면 충돌은 발생하지 않는다. 
  - 때때로 한 데이터센터가 고장 나서 트래픽을 다른 데이터센터로 라우팅해야 하거나 사용자가 다른 지역으로 이동해 현재는 다른 데이터센터가 가깝다면 레코드를 위해 지정된 리더를 변경하고 싶을 때는 충돌 회피가 실패한다.
- **일관된 상태 수렴**
  - 단일 리더 데이터베이스는 마지막 쓰기가 필드의 최종 값으로 결정된다.
  - 다중 리더 설정은 쓰기 순서가 정해지지 않아 최종 값이 무엇인지 명확하지 않다.
  - 각 복제 서버가 쓰기를 본 순서대로 적용한다면 데이터베이스는 일관성 없는 상태가 된다.
  - 데이터베이스는 수렴 방식으로 충돌을 해소해야 한다.
    - 각 쓰기에 고유 ID를 부여하고 가장 높은 ID를 가진 쓰기를 고른다. 타임스탬프를 사용하는 경우를 최종 쓰기 승리라고 한다. 데이터 유실의 위험이 있다.
    - 각 복제 서버에 고유 ID를 부여하고 높은 숫자의 복제 서버에서 생긴 쓰기가 낮은 숫자의 복제 서버에서 생긴 쓰기보다 우선적으로 적용되게 한다. 데이터 유실의 위험이 있다.
    - 명시적 데이터 구조에 충돌을 기록해 모든 정보를 보존하고 나중에 충돌을 해소하는 애플리케이션 코드를 작성한다.
- **사용자 정의 충돌 해소 로직**
  - 충돌을 해소하는 가장 적합한 방법은 애플리케이션에 따라 다르기 때문에 대부분의 다중 리더 복제 도구는 애플리케이션 코드를 사용해 충돌 해소 로직을 작성한다.
    - 쓰기 수행 중
      - 복제된 변경 사항 로그에서 데이터베이스 시스템이 충돌을 감지하자마자 충돌 핸들러를 호출한다.
    - 읽기 수행 중
      - 충돌을 감지하면 모든 충돌 쓰기를 저장한다. 다음 번 데이터를 읽을 때 여러 버전의 데이터가 애플리케이션에 반환되고 충돌 내용을 보여주거나 자동으로 해소할 수 있다.
  - 충돌 해소는 보통 개별 로우나 문서 수준에서 적용되므로 여러 다른 쓰기를 수행하는 트랜잭션이라면 각 쓰기는 충돌 해소를 위해 별도로 간주된다.
  - 자동 충돌 해소
    - 충돌 없는 복제 데이터타입은 셋, 맵, 정렬 목록, 카운터 등을 위한 데이터 구조의 집합으로 합리적인 방법으로 충돌을 자동 해소한다.
    - 병합 가능한 영속 데이터 구조는 깃과 유사하게 명시적으로 히스토리를 추적하고 삼중 병합 함수를 사용한다.
    - 운영 변환은 협업 편집 애플리케이션의 충돌 해소 알고리즘이다.

##### 다중 리더 복제 토폴로지
- 복제 토폴로지는 쓰기를 한 노드에서 다른 노드로 전달하는 통신 경로를 설명한다.
- 가장 일반적인 토폴로지는 전체 연결인데 각자의 쓰기를 다른 모든 리더에서 전송한다.
- 원형 토폴로지는 각 노드가 하나의 노드로부터 쓰기를 받고, 이 쓰기를 다른 한 노드에 전달한다.
- 별 모양 토폴로지는 지정된 루트 노드 하나가 다른 모든 노드에 쓰기를 전달한다. 
- 원형과 별 모양 토폴로지는 여러 노드를 거쳐야 하므로 무한 복제 루프를 방지하기 위해 각 노드에는 고유 식별자가 있고 복제 로그에서 각 쓰기는 거치는 모든 노드의 식별자가 태깅된다.
- 원형과 별 모양 토폴로지는 하나의 노드에 장애가 발생하면 다른 노드 간 복제 메시지 흐름에 방해를 주는데 해당 노드가 복구될 때까지 통신을 할 수 없다.
  - 메시지가 여러 경로를 따라 이동할 수 있으면 단일 장애점을 피할 수 있기 때문에 조금 더 빽빽하게 연결한 토폴로지의 내결함성이 훨씬 더 좋다.
- 전체 연결 토폴로지는 일부 네트워크 연결이 다른 연결보다 빠르면 일부 복제 메시지가 다른 메시지를 추월해 인과성 문제를 야기할 수 있다.
  - 모든 쓰기에 간단히 타임스템프를 추가하는 방식은 노드들의 시간이 동기화됐다고 신뢰할 수 없기 때문에 충분하지 않다.
  - 올바르게 정렬하기 위해 버전 벡터라는 기법을 사용할 수 있다.

#### 리더 없는 복제
- 일부 데이터 저장소 시스템은 리더의 개념을 버리고 모든 복제 서버가 클라이언트로부터 쓰기를 직접 받을 수 있게 허용하는 접근 방식을 사용하기도 한다.
- 아마존이 내부 다이나모 시스템에서 사용 후 유행했는데 이런 종류의 데이터베이스를 다이나모 스타일이라 한다.
##### 노드가 다운됐을 때 데이터베이스에 쓰기
  - 리더 기반 설정에서 쓰기 처리를 계속 하려면 장애 복구를 실행해야 한다.
  - 리더 없는 설정에서는 장애 복구가 필요하지 않고 클라이언트는 복제 서버 중 하나가 쓰기를 놓친 사실을 단순히 무시한다.
  - 사용할 수 없었던 노드가 온라인 상태가 되고 클라이언트가 이 노드에서 읽기를 시작하면 오래된(outdated) 값을 얻을 수 있으므로 하나의 복제 서버로 요청을 보내지 않고 읽기 요청을 병렬로 여러 노드에 전송한다.
    - 버전 숫자를 사용해 어떤 값이 최신 내용인지 결정한다.
- **읽기 복구와 엔트로피**
  - 사용 불가능한 노드가 온라인 상태가 된 후 누락된 쓰기를 따라잡는 방법
    - 읽기 복구
      - 클라이언트가 여러 노드에서 병렬로 읽기를 수행하면 오래된 응답을 감지할 수 있는데 해당 복제 서버에 새로운 값을 다시 기록하는 방식인데 값을 자주 읽는 상황에 적합하다.
    - 안티 엔트로피 처리
      - 백그라운드 프로세스로 복제 서버 간 데이터 차이를 지속적으로 찾아 누락된 데이터를 복사한다. 복사되기까지 상당한 지연이 있을 수 있다.
- **읽기와 쓰기를 위한 정족수**
  - 모든 성공한 쓰기가 모든 세 개의 복제 서버 중 적어도 두 개의 복제 서버에 존재한다는 것을 보장하면 하나의 복제 서버가 오래된 데이터임을 의미하므로 적어도 두 개의 복제 서버에서 읽으면 두 개 중 적어도 하나는 최신 값인지 확인할 수 있다.
  - n개의 복제 서버가 있을 때 모든 쓰기는 w개의 노드에서 성공해야 쓰기가 확정되고 모든 읽기는 최소한 r개의 노드에 질의해야 한다고 가정하면 w+r>n이면 읽을 때 최신 값을 얻을 것으로 기대한다.
  - 일반적으로 n을 홀수로 하고 w = r = (n + 1) / 2로 설정한다.
    - 읽기는 더 빨라지지만 노드 하나가 고장나면 모든 데이터베이스 쓰기가 실패하는 단점이 있다.
  - 정족수 조건이 w + r > n일 때 사용 불가능한 노드 용인
    - w < n이면 노드 하나를 사용할 수 없어도 쓰기를 처리할 수 있다.
    - r < n이면 노드 하나를 사용할 수 없어도 읽기를 처리할 수 있다.
    - n = 3, w = 2, r = 2이면 사용 불가능한 노드 하나를 용인한다.
    - n = 5, w = 3, r = 3이면 사용 불가능한 노드 둘을 용인한다.
  - 필요한 w나 r개 노드보다 사용 가능한 노드가 적다면 쓰기나 읽기는 에러를 반환한다.

##### 정족수 일관성의 한계
- n개의 복제 서버가 있고 w + r > n 이 되게끔 한다면 일반적으로 모든 읽기는 키의 최신 값을 반환할 것을 기대한다.
- 보통 r과 w의 값으로 노드의 과반수를 선택하는데 n/2 노드 장애까지 허용해도 w + r > n이 보장되기 때문이다.
- w + r <= n이 되게끔 설정하면 읽기와 쓰기를 계속 n개의 노드로 전송하지만 성공에 필요한 성공 응답의 수는 더 적다.
- w와 r이 작을수록 오래된 값을 읽을 확률이 높고 낮은 지연 시간과 높은 가용성이 가능하다.
  - 네트워크 중단으로 많은 복제 서버가 응답하지 않는다면 읽기와 쓰기 처리가 계속 진행될 가능성이 높다.
- w + r > n 인 경우에 오래된 값을 반환하는 엣지 케이스
  - 느슨한 정족수를 사용한다면 w개의 쓰기는 r개의 읽기와 다른 노드에서 수행될 수 있으므로 r개의 노드와 w개의 노드가 겹치는 것을 보장하지 않는다.
  - 두 개의 쓰기가 동시에 발생하면 어떤 쓰기가 먼저 일어났는지 분명하지 않다.
  - 쓰기가 읽기와 동시에 발생하면 쓰기는 일부 복제 서버에만 반영될 수 있다.
  - 쓰기가 일부 복제 서버에서는 성공했지만 다른 복제 서버에서 실패해 전체에서 성공한 서버가 w 복제 서버보다 적다면 성공한 복제 서버에서는 롤백하지 않는다.
  - 새 값을 전달하는 노드가 고장나면 예전 값을 가진 복제 서버에서 해당 데이터가 복원되고 새로운 값을 저장한 복제 서버 수가 w보다 낮아져 정족수 조건이 깨진다.
  - 모든 과정이 올바르게 동작해도 시점 문제로 에지 케이스가 있을 수 있다.
- 최종적 일관성을 허용하는 사용 사례에 맞게 최적화됐다.
- **최신성 모니터링**
  - 운영 관점에서 데이터베이스가 최신 결과를 반환하는지 여부를 모니터링하는 일은 중요하다.
  - 복제가 명확히 뒤처진다면 원인을 조사할 수 있게 알려줘야 한다.
  - 리더의 현재 위치에서 팔로워의 현재 위치를 빼면 복제 지연량을 측정할 수 있다.
  - 리더 없는 복제에서는 쓰기가 적용된 순서를 고정할 수 없어 모니터링이 어렵고 특히 읽기 복구만 사용한다면 제한이 없어 아주 오래된 값일 수 있다.
  - 최종적 일관성은 의도적으로 모호한 보장이지만 운용성을 위해서는 최종적을 정량화할 수 있어야 한다.

##### 느슨한 정족수와 암시된 핸드오프
- 정족수는 내결함성이 없어 네트워크 중단으로 다수의 데이터베이스 노드와 클라이언트가 연결이 쉽게 끊어질 수 있다.
- 노드가 n개 이상인 대규모 클러스터에서 네트워크 장애 발생 시 일부 데이터베이스 노드에 연결된다면 트레이드 오프가 생긴다.
  - w나 r 노드 정족수를 만족하지 않는 모든 요청에 오류를 반환한다.
  - 쓰기를 받아들이고 값이 저장되는 n개 노드에 속하지는 않지만 연결할 수 있는 노드에 기록한다. (느슨한 정족수)
- 쓰기와 읽기는 여전히 w와 r의 성공 응답이 필요하지만 값을 위해 지정된 N개의 홈 노드에 엇는 노드가 포함될 수 있다.
- 네트워크 장애 상황이 해제되면 한 노드가 다른 노드를 위해 일시적으로 수용한 모든 쓰기를 해당 홈 노드로 전송하는 방식을 암시된 핸드오프라 부른다.
- 쓰기 가용성을 높이는 데 유용하지만 w + r > n인 경우에도 키의 최신 값을 읽는다고 보장하지 않는다.
- **다중 데이터센터 운영**
  - 리더 없는 복제도 동시 쓰기 충돌, 네트워크 중단, 지연 시간 급증을 허용하기 때문에 다중 데이터센터 운영에 적합하다.
    - 모든 복제 서버에 전송되지만, 대게 비동기로 방생하게끔 설정한다.

##### 동시 쓰기 감지
- 다이나모 스타일 데이터베이스는 여러 클라이언트가 동시에 같은 키에 쓰는 것을 허용하기 때문에 엄격한 정족수를 사용하더라도 충돌이 발생한다.
- 다양한 네트워크 지연과 부분적인 장애 때문에 이벤트가 다른 노드에 다른 순서로 도착할 수 있다는 문제가 있다.
- 최종적인 일관성을 달성하기 위해 복제본들은 동일한 값이 돼야한다.
- **최종 쓰기 승리(동시 쓰기 버리기)**
  - 가장 최신 값으로 덮어써서 동일한 값으로 수렴하게 한다.
  - 타임스탬프를 붙여 최신이라는 것을 임의로 정할 수 있다.
  - LWW로 데이터베이스를 안전하게 사용하는 유일한 방법은 키를 한번만 쓰고 이후에는 불변 값으로 다루는 것이다.
- **"이전 발생" 관계와 동시성**
  - 작업 B가 작업 A에 대해서 알거나 A에 의존적이거나 어떤 방식으로든 A를 기반으로 한다면 작업 A는 작업 B의 이전 발생이다.
  - 작업이 동시에 발생하면 동시 작업이라 한다.
- **이전 발생 관계 파악하기**
  - 서버는 버전 번호를 보고 두 작업이 동시에 수행됐는지 여부를 결정할 수 있으므로 값 자체를 해석할 필요 없고 값을 데이터 구조로 사용할 수 있다.
    - 클라이언트가 키를 읽을 때는 덮어쓰지 않은 모든 값을 반환한다.
    - 클라이언트가 키를 기록할 때는 이전 읽기의 버전 번호를 포함해야 하고 이전 읽기에서 받은 모든 값을 함께 합쳐야 한다.
    - 서버가 특정 버전 번호를 가진 쓰기를 받을 때 해당 버전 이하 모든 값을 덮어쓸 수 있지만 높은 버전 번호의 모든 값은 유지해야 한다.
  - 버전 번호를 포함하지 않은 쓰기는 다른 쓰기와 동시에 수행된 것이므로 아무것도 덮어쓰지 않는다.
- **동시에 쓴 값 병합**
  - 여러 작업이 동시에 발생하면 클라이언트는 동시에 쓴 값을 합쳐 정리해야 하는데 이런 동시 값을 형제 값이라 부른다.
  - 형제 값의 병합은 다중 리더 복제에서 충돌을 해소하는 문제와 본질적으로 같다.
  - 형제를 병합하는 합리적인 접근 방식은 합집합을 취하는 것이다.
  - 상품을 제거할 때는 제거했음을 나타내기 위해 해당 버전 번호에 표시를 남겨둬야 하는데 이를 툼스톤이라 한다.
- **버전 벡터**
  - 버전 번호는 다중 복제본의 동시 쓰기를 받아들일 때는 충분하지 않다.
    - 복제본당 버전 번호도 사용해야 한다.
  - 모든 복제본의 버전 번호 모음을 버전 벡터라고 부른다.
  - 값을 읽을 때 데이터베이스 복제본에서 클라이언트로 보낸다.

#### 정리
- 소프트웨어 버그 때문에 드러나지 않는 데이터 오염과 같이 모르는 사이에 일어나는 결함까지는 고려하지 않더라도 최소한 사용할 수 없는 노드와 네트워크 중단에는 대처해야 한다.

### 💡 느낀 점
- 충돌을 해소하는 방식에서 모든 정보를 보존하고 나중에 충돌을 해소하는 애플리케이션 코드를 작성하는 방법이 있었는데 이것이 깃에서 충돌을 처리하는 방식이라는 생각이 들었다.
- 분산 시스템에서 동시성과 충돌 문제는 항상 발생하는 일이고 시스템에 따라 어떤 방식을 선택할 지 트레이드 오프를 고려해 결정해야겠다는 생각이 들었다.
