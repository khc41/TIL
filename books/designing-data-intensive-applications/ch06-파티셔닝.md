### 6장. 파티셔닝
- 데이터셋이 매우 크거나 질의 처리량이 매우 높다면 복제만으로는 부족하고 데이터를 파티션으로 쪼갤 필요가 있는데 이 작업을 샤딩이라고도 한다.
- 파티션을 나눌 때는 보통 각 데이터 단위가 하나의 파티션에 속하게 한다.
- 파티셔닝은 확장성을 위해 사용되는데 대용량 데이터셋이 여러 디스크에 분산될 수 있고 질의 부하는 여러 프로세서에 분산될 수 있다.
- 각 노드에서 자신의 파티션에 해당하는 질의를 독립적으로 실행할 수 있으므로 노드를 추가함으로써 질의 처리량을 늘릴 수 있다.

#### 파티셔닝과 복제
- 보통 복제와 파티셔닝을 함께 적용해 각 파티션의 복사본을 여러 노드에 저장해서 내결함성을 보장할 수 있다.
- 각 노드는 어떤 파티션에서는 리더이면서 다른 파티션에게는 팔로워가 될 수 있다.

#### 키-값 데이터 파티셔닝
- 파티셔닝의 목적은 질의 부하를 노드 사이에 고르게 분산시키는 것이다.
- 파티셔닝이 고르게 이뤄지지 않아 다른 파티션보다 데이터가 많거나 질의를 많이 받는 파티션이 있다면 쏠렸다(skewed)라고 말한다.
- 불균형하게 부하가 높은 파티션을 핫스팟이라고 한다.
- 핫스팟을 회피하는 가장 단순한 방법은 레코드를 할당할 때 노드를 무작위로 선택하는 것이지만, 어떤 노드에 저장됐는지 알 수 없으므로 모든 노드에서 병렬적으로 질의를 실행해야 한다.

##### 키 범위 기준 파티셔닝
- 각 파티션에 연속된 범위의 키(최솟값에서 최댓값까지)를 할당하는 것이다.
- 데이터가 고르게 분포하지 않을 수 있기 때문에 키 범위 크기가 반드시 동일할 필요는 없다.
- 각 파티션 내에서는 키를 정렬된 순서로 저장할 수 있는데 이는 범위 스캔이 쉬워지고 질의 하나로 관련 레코드 여러 개를 읽어오는 데 사용할 수 있다.
- 특정한 접근 패턴이 핫스팟을 유발하는 단점이 있다.

##### 키의 해시값 기준 파티셔닝
- 쏠림과 핫스팟의 위험 때문에 많은 분산 데이터스토어는 키의 파티션을 정하는 데 해시 함수를 사용한다.
- 좋은 해시 함수는 쏠린 데이터를 입력으로 받아 균일하게 분산되게 한다.
- 키에 적합한 해시 함수를 구했다면 각 파티션에 해시값 범위를 할당하고 해시값이 파티션의 범위에 속하는 모든 키를 그 파티션에 할당하면 된다.
- 일관성 해싱
  - CDN 같은 인터넷 규모의 캐시 시스템에서 부하를 균등하게 분산시키는 방법이다.
  - 중앙 제어나 분산 합의가 필요하지 않도록 파티션 경계를 무작위로 선택한다.
- 파티셔닝 키에 해시값을 사용하면 범위 질의를 효율적으로 실행할 수 없다.

##### 쏠린 작업부하와 핫스팟 완화
- 키를 해싱하는 방법도 동일한 키를 읽고 쓰는 극단적인 상황에서는 모든 요청이 동일한 파티션으로 쏠려 핫스팟이 발생한다.
- 현대 데이터 시스템은 쏠린 작업부하를 자동으로 보정하지 못하므로 애플리케이션에서 쏠림을 완화해야 한다. 
  - 각 키의 시작이나 끝에 임의의 숫자를 붙여 다른 키로 분산시킬 수 있다.
  - 하지만 이 방법은 읽기를 실행할 때 추가적인 작업이 필요해진다.
  - 요청이 몰리는 소수의 키에만 적용하는 게 타당하다.

#### 파티셔닝과 보조 색인
- 보조 색인은 특정한 값이 발생한 항목을 검색하는 수단으로 파티셔닝과 함께하면 복잡해진다.
- 많은 키-값 저장소에서는 구현 복잡도가 추가되는 것을 피하려고 보조 색인을 지원하지 않는다.

##### 문서 기준 보조 색인 파티셔닝
- 각 항목에는 문서ID라고 부르는 고유 ID가 있고 데이터베이스를 문서 ID 기준으로 파티셔닝 한다고 가정한다.
- 보조 색인을 만들 때 색인 항목에 해당하는 문서 ID를 목록에 추가하면 각 파티션이 완전히 독립적으로 동작하는데 이 방식을 지역 색인이라고 한다.
- 읽을 때는 모든 파티션으로 질의를 보내서 얻은 결과를 모두 모아야 한다. 
- 파티셔닝된 데이터베이스에 이런 식으로 질의를 보내는 방식을 스캐터/게더 라고도 하는데 보조 색인을 써서 읽는 질의는 큰 비용이 들 수 있다.

##### 용어 기준 보조 색인 파티셔닝
- 모든 파티션의 데이터를 담당하는 전역 색인을 만들 수도 있다.
- 노드가 병목이 되는 것을 막기 위해 한 노드에만 색인을 저장하지 않는다.
- 찾고자하는 용어에 따라 저장되는 파티션이 결정되므로 용어 기준으로 파티셔닝됐다고 한다.
- 전역 색인이 갖는 이점은 읽기가 효율적이지만 쓰기가 느리고 복잡하다.
- 현실에서는 전역 보조 색인은 대개 비동기로 갱신되는데, 쓰기를 실행한 후 바로 색인을 읽으면 변경 사항이 색인에 반영되지 않았을 수도 있다.


#### 파티션 재균형화
- 클러스터에서 한 노드가 담당하던 부하를 다른 노드로 옮기는 과정을 재균형화라고 한다.
- 최소 요구사항
  - 재균형화 후 부하가 노드들 사이에 균등하게 분배돼야 한다.
  - 재균형화 도중에도 읽기 쓰기 요청을 받아들여야 한다.
  - 재균형화가 빨리 실행되고 네트워크와 디스크 I/O 부하를 최소화할 수 있도록 데이터가 필요 이상으로 옮겨져서는 안 된다.

##### 재균형화 전략
- **쓰면 안 되는 방법: 해시값에 모드 N 연산을 실행**
  - 노드 개수 N이 바뀌면 대부분의 키가 노드 사이에 옮겨져야 해서 재균형화 비용이 지나치게 커진다.
- **파티션 개수 고정**
  - 파티션을 노드 대수보다 많이 만들고 각 노드에 여러 파티션을 할당하는 것
  - 노드가 추가되면 새 노드는 파티션이 균일하게 분배될 때까지 기존 노드에서 파티션 몇 개를 뺏어올 수 있다.
  - 파티션은 노드 사이에서 통째로 이동하기만 한다.
  - 파티션 할당 변경은 네트워크를 통해 대량의 데이터를 전송해야 하므로 전송이 진행중인 동안엔 읽기나 쓰기가 기존에 할당된 파티션을 통해 실행된다.
  - 성능이 좋은 노드에 파티션을 더 할당함으로써 더 많은 부하를 담당하게 할 수 있다.
  - 처음 설정된 파티션 개수가 사용 가능한 노드 대수의 최대치가 되므로 미래에 증가될 것을 수용하기에 충분히 높은 값을 선택해야 하는데 개별 파티션도 관리 오버헤드가 있으므로 너무 큰 수를 선택하면 역효과를 낳을 수 있다.
  - 각 파티션에는 전체 데이터의 고정된 비율이 포함되므로 개별 파티션 크기는 클러스터의 전체 데이터 크기에 비례해서 증가한다.
- **동적 파티셔닝**
  - 키 범위 파티셔닝을 사용하는 경우에는 파티션 경계를 잘못 지정하면 모든 데이터가 한 파티션에 저장되고 나머지는 텅 빌 수 있다.
  - 파티션 크기가 설정된 값을 넘어서면 파티션을 두 개로 쪼개 각가에 원래 파티션의 절반 정도의 데이터가 포함되게 한다.
  - 데이터가 많이 삭제되어 임곗값 아래로 떨어지면 인접한 파티션과 합쳐질 수 있다.
  - 큰 파티션이 쪼개진 후 부하의 균형을 맞추기 위해 분할된 파티션 중 하나가 다른 노드로 이동될 수 있다.
  - 파티션 개수가 전체 데이터 용량에 맞춰 조정되므로 데이터 양이 작으면 파티션 개수가 적어도 되므로 오버헤드도 작다.
  - 빈 데이터베이스는 사전 정보가 없으므로 시작할 때는 파티션이 하나이다.

- **노드 비례 파티셔닝**
  - 노드당 할당되는 파티션 개수를 고정한다.
    - 노드 대수가 변함 없을 동안 파티션 크기는 데이터셋 크기에 비례해 증가하지만, 노드 대수를 늘리면 파티션 크기는 작아진다.
  - 데이터 용량이 클수록 저장할 노드도 많이 필요하므로 개별 파티션 크기도 상당히 안정적으로 유지된다.
  - 새 노드가 클러스터에 추가되면 고정된 개수의 파티션을 무작위로 선택해 분할하고 각 분할된 파티션의 절반은 그대로 두고 다른 절반은 새 노드에 할당한다.

##### 운영: 자동 재균형화와 수동 재균형화
- 완전 자동 재균형화는 일상적인 유지보수에 편리하지만 예측하기 어렵다.
  - 네트워크나 노드에 과부화가 걸릴 수 있고 재균형화 진행중인 동안에 다른 요청의 성능이 저하될 수 있다.
  - 자동 장애 감지와 조합되면 부하를 더해 연쇄 장애가 발생할 가능성이 있다.

#### 요청 라우팅
- 어느 노드로 접속해야 하는지의 문제는 서비스 찾기(service discovery)의 일종이다.
- 접근법
  - 클라이언트가 아무 노드에 접속하게 한다.
  - 모든 요청을 라우팅 계층으로 보낸다.
  - 클라이언트가 파티셔닝 방법과 파티션이 어떤 노드에 할당됐는지를 알고 있게 한다.
- 많은 분산 데이터 시스템은 클러스터 메타데이터를 추적하기 위해 주키퍼 같은 별도의 코디네이션 서비스를 사용한다.
  - 노드가 추가되거나 삭제되면 라우팅 계층에 이를 알려서 라우팅 정보를 최신으로 유지한다.
- 가십 프로토콜은 아무 노드나 요청을 받고 요청을 받은 노드는 요청을 처리할 파티션을 갖고 있는 올바른 노드로 요청을 전달한다.

##### 병렬 질의 실행
- 분석용으로 사용되는 데이터베이스는 대규모 병렬 처리가 가능하다.
- 복잡한 질의를 여러 실행 단계와 파티션으로 분해하며 서로 다른 노드 내에서 병렬적으로 실행될 수 있다.

#### 정리
- 파티셔닝의 목적은 핫스팟이 생기지 않게 하면서 데이터와 질의 부하를 여러 장비에 균일하게 분배하는 것이다.

#### 느낀점
- 데이터베이스가 파티셔닝을 통해 여러 노드와 파티션으로 분배되면서 어떤 노드와 어떤 파티션에서 데이터를 가져오는지, 데이터의 재분배는 어떤식으로 하는지를 배울 수 있었다.
- 이렇게 분산 데이터베이스는 성능상의 이점은 많고 고가용성을 유지하기에 용이하지만 시스템의 복잡도가 많이 올라가서 도입할 때 충분히 고려하지 않으면 장애가 더 많이 날 것 같다는 생각이 들었다.
- 데이터베이스가 복제되면서 네트워크를 통해서 복제될텐데 네트워크는 항상 연결이 지연되거나 끊기는 상황을 고려해야한다는 것을 다시금 느꼈다.
