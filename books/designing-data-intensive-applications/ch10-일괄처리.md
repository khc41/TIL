### 10장. 일괄 처리
- 시스템의 세 가지 유형
  - 서비스(온라인 시스템): 클라이언트로부터 요청을 받아 가능한 빨리 요청을 처리한다. 응답 시간이 중요하다.
  - 일괄 처리 시스템(오프라인 시스템): 매우 큰 입력 데이터를 받아 작업을 수행하고 결과 데이터를 생산한다. 처리량이 중요하다.
  - 스트림 처리 시스템(준실시간 시스템): 입력 데이터를 소비하고 출력 데이터를 생산하지만 입력 이벤트가 발생한 직후 바로 작동한다. 
- 일괄 처리는 신뢰할 수 있고 확장 가능하며 유지보수하기 쉬운 애플리케이션을 구축하는 데 매우 중요한 구성요소다.

#### 유닉스 도구로 일괄 처리하기
##### 단순 로그 분석
##### 유닉스 철학
- 각 프로그램이 한 가지 일만 하도록 작성하라.
- 모든 프로그램의 출력은 아직 알려지지 않은 다른 프로그램의 입력으로 쓰일 수 있다고 생각하라. 불필요한 정보로 출력이 오염되지 않도록 하라.
- 소프트웨어를 빠르게 써볼 수 있게 설계하고 구축하라.
- 프로그래밍 작업을 줄이려면 미숙한 도움보단 도구를 사용하라.
- 자동화, 빠른 프로토타이핑, 증분 반복, 실험 친화, 큰 프로젝트를 청크로 나누어 처리하기와 같은 방법은 오늘날 애자일 및 데브옵스 운동과 매우 흡사하다.
- **동일 인터페이스**
  - 출력을 다른 프로그램의 입력으로 쓰기 위해 모두가 같은 입출력 인터페이스를 사용해야 한다.
- **로직과 연결의 분리**
  - 프로그램이 특정 파일의 경로가 어디에 있는지 신경쓰지 않고 stdin과 stdout만으로 처리하고 싶다면 유닉스 접근법이 가장 좋다.
  - 프로그램은 입력이 어디서부터 들어오는지 출력이 어디로 나가는지 신경 쓰거나 알 필요조차 없는데 이런 형태를 느슨한 결합, 지연 바인딩, 제어 반전이라 한다.
- **투명성과 실험**
  - 유닉스 도구는 진행 사항을 파악하기 상당히 쉽다.
    - 입력 파일은 불변으로 처리된다.
    - 어느 시점이든 파이프라인을 중단하고 출력을 파이프를 통해 less로 보내 원하는 형태의 출력이 나오는 지 확인 가능하다.
    - 특정 파이프라인 단계의 출력을 파일에 쓰고 그 파일을 다음 단계의 입력으로 사용할 수 있다.

#### 맵리듀스와 분산 파일 시스템
- 유닉스 도구와 비슷한 면이 있지만 수천 대의 장비로 분산해서 실행 가능하다.
- HDFS (Hadoop Distributed File System)는 비공유 원칙을 기반으로 하는데 NAS(Network Attached Storage) 와 SAN(Storage Area Network) 아키텍처에서 사용하는 공유 디스크 방식과는 반대다.
  - 특별한 하드웨어가 필요없고 데이터센터 네트워크에 연결된 컴퓨터면 충분하다.
- 각 장비에서 실행된 데몬 프로세스로 구성되고 네임노드라고 부르는 중앙 서버는 특정 파일 블록이 어떤 장비에 저장됐는지 추적해 개념적으로 매우 큰 하나의 파일 시스템이고 데몬이 실행 중인 모든 장비의 디스크를 사용할 수 있다.
- 장비가 죽거나 디스크가 실패하는 경우에 대비하기 위해 파일 블록은 여러 장비에 복제된다. RAID와 유사하지만 파일의 접근과 복제가 특별한 하드웨어 장치 없이 평범한 데이터 센터 네트워크 상에서 이뤄진다는 점이 다르다.
- HDFS는 확장성이 뛰어나며 데이터 저장과 접근 비용은 범용 하드웨어와 오픈소스 소프트웨어를 사용하기 때문에 동급 용량의 전용 저장소 장치를 사용하는 비용보다 훨씬 저렴하다.

##### 맵리듀스 작업 실행하기
- 맵리듀스의 데이터 처리 패턴은 로그 분석 예제와 상당히 비슷하다.
  1. 입력 파일을 읽고 레코드로 쪼갠다.
  2. 각 입력 레코드마다 매퍼 함수를 호출해 키와 값을 추출한다. 
  3. 키를 기준으로 키-값 쌍을 모두 정렬한다. 
  4. 정렬된 키-값 쌍 전체를 대상으로 리듀스 함수를 호출한다. 같은 키가 여러번 등장했다면 정렬 과정에서 서로 인접하므로 메모리 상에 상태를 유지하지 않고도 쉽게 결합할 수 있다.
- 매퍼
  - 모든 입력 레코드마다 한 번씩 호출되고 입력 레코드로부터 키와 값을 추출한다.
- 리듀서
  - 매퍼가 생산한 키-값 쌍을 받아 같은 키를 가진 레코드를 모으고 해당 값의 집합을 반복해 리듀서 함수를 호출해 출력 레코드를 생산한다.
- **맵리듀스의 분산 실행**
  - 맵리듀스는 병렬로 수행하는 코드를 직접 작성하지 않고도 여러 장비에서 동시에 처리가 가능하다.
  - 맵리듀스 프레임워크가 장비 간에 데이터가 이동하는 복잡한 부분을 처리한다.
  - 맵리듀스의 작업 병렬실행은 파티셔닝을 기반으로 한다.
  - 각 매퍼 입력 파일의 복제본이 있는 장비에 RAM과 CPU에 여유가 충분하다면 맵리듀스 스케줄러가 입력 파일이 있는 장비에서 작업을 수행하려 하는데 이 원리를 데이터 가까이에서 연산하기라 하고 네트워크를 통해 입력 파일을 복사하는 부담과 네트워크 부하가 감소하고 지역성이 증가한다.
  - 맵리듀스 프레임워크가 작업을 수행하기 적절한 장비로 코드를 복사하고 매퍼 태스크가 시작된다. 
  - 같은 키를 가진 모든 키-값 쌍을 같은 리듀서에서 처리하는 것을 보장하는데, 특정 키-값 쌍이 어느 리듀스 태스크에서 수행될지 결정하기 위해 키의 해시값을 사용한다.
  - 데이터셋이 매우 크기 때문에 일반적인 정렬 알고리즘으로 한 장비에서 모두 정렬하기는 쉽지 않아 단계를 나누어 정렬을 수행한다. 
    - 맵 태스크는 키의 해시값을 기반으로 출력을 리듀서로 파티 셔닝하고 로컬 디스크에 정렬된 파일로 기록한다. 그 후 맵리듀스 스케줄러는 그 매퍼에서 출력파일을 가져올 수 있다고 리듀서에 알려준다.
    - 리듀서는 각 매퍼와 연결해서 담당하는 파티션에 해당하는 정렬된 키-값 쌍 파일을 다운로드하는데 데이터 파티션을 복사하는 과정을 셔플이라 한다. 
- **맵리듀스 워크플로**
  - 작업 하나로 해결할 수 있는 문제의 범위는 제한적이라 일반적으로 작업을 연결해 워크플로로 구성한다. 
  - 연결된 맵리듀스 작업은 각 명령의 출력을 임시 파일에 쓰고 다음 명령이 그 임시 파일로부터 입력을 읽는 방식에 가깝다.

##### 리듀스 사이드 조인과 그룹화
- 맵리듀스는 색인 개념이 없고 대량 레코드 대상으로 집계 연산을 하므로 입력 전체를 스캔하는게 합리적이다.
- **사용자 활동 이벤트 분석 예제**
  - 사용자 활동 이벤트는 사용자 프로필 정보를 가지고 있지 않고 ID만 포함하기 때문에 사용자 프로필 데이터베이스를 조인해야 한다.
  - 데이터베이스의 사본을 가져와 사용자 활동 이벤트 로그가 저장된 분산 파일 시스템에 넣으면 같은 HDFS 상에 존재하고 맵리듀스를 사용해 효율적인 처리가 가능하다.
- **정렬 병합 조인**
  - 맵리듀스 프레임워크에서 키-값 쌍으로 정렬한다면 사용자의 활동 이벤트와 사용자 레코드는 리듀서의 입력으로 서로 인접해서 들어간다.
  - 리듀서가 항상 사용자 데이터베이스를 먼저 보고 활동 이벤트를 시간 순으로 보게하는 식으로 맵리듀스에서 작업 레코드를 재배열하기도 하는데 이 기술을 보조 정렬이라 한다.
  - 리듀서는 특정 사용자 ID의 모든 레코드를 한 번에 처리하므로 한 번에 사용자 한 명의 레코드만 메모리에 유지하면 되고 네트워크로 아무 요청도 보낼 필요가 없는데 이 알고리즘을 정렬 병합 조인이라고 한다.
- **같은 곳으로 연관된 데이터 가져오기**
  - 맵리듀스 프로그래밍 모델은 올바른 장비로 데이터를 모으는 연산의 물리적 네트워크 통신 측면과 받은 데이터를 처리하는 애플리케이션 로직을 분리한다.
  - 맵리듀스는 모든 네트워크 통신을 직접 관리하기 때문에 특정 장비가 죽는 것과 같이 부분적으로 실패가 발생하더라도 애플리케이션 코드 단에서 고민할 필요가 없다.
- **그룹화**
  - 맵리듀스로 그룹화 연산을 구현하는 가장 간단한 방법은 매퍼가 키-값 쌍을 생성할 때 그룹화할 대상을 키로 하는 것이다.
  - 특정 사용자가 취한 일련의 활동을 찾기 위해 사용자 세션별 활동 이벤트를 수집 분석할 때도 일반 적으로 그룹화를 사용하는데 이를 세션화라 한다.
- **쏠림 다루기**
  - 키 하나에 너무 많은 데이터가 연관된다면 같은 키를 가지는 모든 레코드를 같은 장소로 모으는 패턴은 제대로 동작하지 않은데 이를 린치핀 객체 또는 핫 키라 한다.
  - 한 리듀서에 모은다면 상당한 쏠림 현상이 생기고 모든 매퍼와 리듀서가 완전히 끝나야지만 맵리듀스 작업이 끝나기 때문에 가장 느린 리듀서가 작업을 완료할 때까지 기다려야 한다.
  - 쏠린 조인 메서드는 핫 키를 결정하기 위해 샘플링 작업을 수행하고 실제 조인을 수행할 때 매퍼는 핫 키를 가진 레코드는 여러 리듀서 중 임의로 선택한 하나로 레코드를 보낸다.
    - 핫 키를 여러 리듀서에 퍼트려서 처리하게 하는 방법으로 다른 조인 입력을 여러 리듀서로 복제하는 비용이 들지만 병렬화 효과가 훨씬 크다.
  - 크런치에서 제공하는 공유 조인 메서드는 비슷하지만 샘플링 작업 대신 핫 키를 명시적으로 지정해야 한다. 
  - 하이브는 핫 키를 메타데이터에 명시적으로 지정하고 핫 키와 관련된 레코드를 나머지 키와 별도 파일에 저장한 뒤 핫 키를 가지는 레코드는 맵 사이드 조인을 사용해 처리한다.  

##### 맵 사이드 조인
- 조인 로직을 리듀서에서 수행하면 리듀스 사이드 조인이라고 한다. 이는 입력 데이터에 대한 특정 가정이 필요 없다. 그러나 정렬 후 리듀서로 복사한 뒤 리듀서 입력을 병합하는 모든 과정에서 드는 비용이 상당하다.
- 입력 데이터에 대해 특정 가정이 가능하다면 맵사이드 조인으로 불리는 기법을 사용해 조인을 더 빠르게 수행할 수 있다.
- **브로드캐스트 해시 조인**
  - 사용자 데이터베이스를 읽어 인메모리 해시 테이블에 넣으면 매퍼는 사용자 활동 이벤트를 모두 스캔할 수 있고 각 이벤트의 사용자 ID를 해시 테이블에서 간단하게 조회할 수 있는데 이 알고리즘을 브로드캐스트 해시 조인이라 한다.
  - 작은 조인 입력을 인메모리 해시 테이블로 적재하는 대신 로컬 디스크에 읽기 전용 색인으로 작은 조인 입력을 저장하기도 한다.
- **파티션 해시 조인**
  - 맵 사이드 조인의 입력을 파티셔닝한다면 해시 조인 접근법을 각 파티션에 독립적으로 적용할 수 있다. 
  - 제대로 파시셔닝이 동작했다면 조인할 레코드 모두가 같은 번호의 파티션에 위치해서 각 매퍼는 각 입력 데이터셋 중 파티션 한 개만 읽어도 충분하다. 
  - 각 매퍼의 해시 테이블에 적재해야 할 데이터의 양을 줄일 수 있다는 점이 장점이다.
  - 파티션 해시 조인은 조인할 두 입력 모두를 같은 키와 같은 해시 함수를 기반으로 같은 수로 파티셔닝해야 작동한다.
- **맵 사이드 병합 조인**
  - 입력 데이터셋이 같은 방식으로 파티셔닝됐을 뿐 아니라 같은 키를 기준으로 정렬됐다면 변형된 맵 사이드 조인을 적용할 수 있다.
  - 파티셔닝 및 정렬된 데이터셋이 바로 조인 외에 다른 용도로 필요한 경우 맵 전용 작업에서 병합 조인을 수행하는 편이 나을 때도 있다.
- **맵 사이드 조인을 사용하는 맵리듀스 워크플로**
  - 맵 사이드 조인을 사용할지 리듀스 조인을 사용할지에 따라 출력 구조가 달라진다. 리듀스 사이드 조인은 조인 키로 파티셔닝하고 정렬해서 출력하고 맵 사이드 조인은 큰 입력과 동일한 방법으로 파티셔닝하고 정렬하낟.
  - 조인 전략을 최적화할 때는 파티션 수가 몇개인지, 데이터가 어떤 키를 기준으로 파티셔닝되고 정렬됐는지 등 분산 파일 시스템 내 저장된 데이터셋의 물리적 레이아웃 파악이 중요하다.

##### 일괄 처리 워크플로의 출력
- 일괄처리는 입력 데이터셋 대부분을 스캔해 분석에 가깝지만 맵리듀스 작업의 워크플로는 분석 목적으로 사용하는 SQL 질의와는 다르다.
- **검색 색인 구축**
  - 맵리듀스는 구글에서 검색 엔진에 사용할 색인을 구축하기 위해 처음 사용됐는데 그때 사용된 워크플로는 5~10개 맵리듀스 작업으로 구현됐다.
  - 정해진 문서 집합을 대상으로 전문 검색이 필요할 때 일괄 처리가 색인을 구축하는 데 매우 효율적이다.
    - 매퍼는 필요에 따라 문서 집합을 파티셔닝한다.
    - 각 리듀서가 해당 파티션에 대한 색인을 구축한다.
    - 색인 파일은 분산 파일 시스템에 저장된다. 
  - 문서 기준으로 파티셔닝해 색인 구축하는 과정은 병렬화가 매우 잘 된다.
  - 색인된 문서 집합이 변하면 전체 문서 집합을 대ㅏㅅㅇ으로 주기적으로 전체 색인 워크플로를 재수행하던가 증분 색인을 구축하는 것이 가능하다.
- **일괄 처리의 출력으로 키-값을 저장**
  - 분류기 같은 머신러닝 시스템을 구축하거나 추천 시스템을 구축할 수 있다.
  - 일괄 처리의 출력은 흔히 일종의 데이터베이스가 돼서 웹 애플리케이션이 질의하는 데이터베이스로 보내야하는데 직접 매퍼와 리듀서 내에서 선호하는 데이터베이스 클라이언트 라이브러리를 사용해 일괄 처리 작업이 한번에 레코드 하나씩 데이터베이스 서버로 직접 요청을 보내는 방법이 있다.
    - 이는 유효한 방법이 아니다. 모든 레코드마다 네트워크 요청을 하는 작업은 일반적인 일괄 처리 태스크의 처리량에 비해 상당히 느리고, 동시에 많은 태스크를 실행하므로 데이터베이스가 과부하 상태에 빠지기 쉽고, 작업이 실패해 재시도하면 부분적으로 완료될 수 있으므로 신경써야한다.
  - 일괄 처리 작업 내부에 완전히 새로운 데이터베이스를 구축해 분산 파일 시스템의 작업 출력 디렉터리에 저장하는 방법이 훨씬 좋다.
- **일괄 처리 출력에 관한 철학**
  - 맵리듀스 작업은 입력을 불변으로 처리하고 외부 데이터베이스에 기록하는 등의 부수 효과를 피하기 때문에 일괄 처리 작업은 좋은 성능을 내면서도 유지보수가 훨씬 간단하다.

##### 하둡과 분산 데이터베이스의 비교
- 대규모 병렬 처리(massively parallel processing, MPP) 데이터베이스와 맵리듀스의 가장 큰 차이는 MPP는 장비 클러스터에서 분석 SQL 질의를 병렬로 수행하는 것에 초점을 두지만 맵리듀스와 분산 파일 시스템의 조합은 아무 프로그램이나 실행할 수 있는 운영체제와 비슷한 속성을 제공한다.
- **저장소의 다양성**
  - 데이터베이스는 특정 모델을 따라 데이터를 구조화해야 하지만 분산 파일시스템의 파일은 어떤 데이터 모델과 인코딩을 사용해서도 기록할 수 있는 연속된 바이트일 뿐이다.
  - 하둡은 데이터가 어떤 형태라도 상관없이 HDFS로 덤프할 수 있는 가능성을 열어 놓지만, MPP 데이터베이스는 데이터베이스에 특화된 저장 형태로 데이터를 가져오기 전에 데이터와 질의 형태를 신중하게 선행 모델링해야 한다.
  - 현실에서는 이상적인 데이터 모델을 만들려고 하기보다 데이터를 빨리 사용 가능하게 만드는 것이 더 가치 있다.
  - MPP 데이터베이스가 요구하는 세심한 스키마 설계는 중앙 집중식 데이터 수집을 느리게 만든다.
  - 제약없는 데이터 덤핑은 데이터를 해석하는 부담을 이전시킨다.
  - 하나의 이상적인 데이터 모델은 존재하지 않을지 몰라도 데이터에는 여러 목적에 적합한 다양한 관점이 존재해 원시 상태로 데이터를 덤프하는 것만으로도 여러 변환이 가능한데, 이런 접근법을 초밥 원리라 부른다.
  - 하둡은 ETL 프로세스를 구현하는 데 종종 사용되기도 하는데 트랜잭션 처리 시스템에서 데이터를 원시 형태로 분산 파일 시스템에 덤프하고 맵리듀스 작업은 데이터를 정리하고 관계형으로 변환한 후 데이터 분석을 위해 MPP 데이터웨어 하우스로 옮긴다. 
- **처리 모델의 다양성**
  - MPP 데이터베이스는 모놀리틱 구조로서 디스크 저장소 레이아웃과 질의 계획, 스케줄링과 실행을 다루는 소프트웨어 조각들이 긴밀하게 통합돼서 설계된 질의 유형에 대해서는 매우 좋은 성능을 얻을 수 있다.
  - 머신러닝이나 추천 시스템 또는 관련성 랭킹 모델을 탑재한 전문 검색 색인을 구축하거나 이미지 분석을 한다면 특정 애플리케이션에 한정되는 경우가 많아 단순한 질의 작성이 아닌 코드 작성이 반드시 필요하다.
  - 맵리듀스를 이용하면 엔지니어는 자신이 작성한 코드를 대용량 데이터셋 상에서 쉽게 실행할 수 있다.
  - 다른 종류의 처리를 하기 위해 다른 특정 시스템으로 데이터를 옮길 필요 없고 새로운 처리 모델을 사용한 실험도 훨씬 쉽다.
- **빈번하게 발생하는 결함을 줄이는 설계**
  - 일괄 처리는 온라인 시스템에 비해 결함에 덜 민감하지만, MPP 데이터베이스는 질의 실행 중에 한 장비만 죽어도 전체 질의가 대부분 중단돼 사용자가 질의를 다시 제출하든지 자동으로 재실행하게 해야한다.
  - MPP 데이터베이스는 디스크에서 데이터를 읽는 비용을 피하기 위해 해시 조인 같은 방식을 사용해 가능하면 메모리에 많은 데이터를 유지하는 것을 선호한다.
  - 맵리듀스는 맵 또는 리듀스 태스크의 실패를 견딜 수 있고, 내결함성을 확보하고 메모리에 올리기 데이터셋이 너무커서 데이터를 되도록 디스크에 기록하려 한다.
  - 맵리듀스가 메모리를 아껴 쓰고 태스크 수준에서 복구하는 이유는 설계할 때 각 태스크 별로 컨테이너를 사용해 리소스를 할당받는데, 우선순위가 높은 태스크에 더 많은 자원이 필요하면 낮은 우선순위인 맵리듀스가 종료될 수 있어서 태스크 종료가 예상치 못하게 자주 발생하더라도 견딜 수 있게 설계됐다.

#### 맵리듀스를 넘어
- 맵리듀스는 학습하기가 매우 유용한 도구로서 분산 파일 시스템 상에서 상당히 단순 명료하게 추상화된 모델이다.
- 맵리듀스를 직접 사용하는 일은 어렵기 때문에 맵리듀스 상에서 추상화된 다양한 고수준 프로그래밍 모델이 등장했다.
- 맵리듀스 실행 모델 자체에도 문제가 있고 일부 유형의 처리에 대해서는 성능 저하를 유발하기도 하지만 맵리듀스는 매우 견고하다.

##### 중간 상태 구체화
- 대개 한 작업의 출력은 같은 팀 내에서 유지보수하는 다른 특정 작업의 입력으로만 사용되는데 이 경우에 분산 파일 시스템 상에 있는 파일들은 단순히 한 작업에서 다른 작업으로 데이터를 옮기는 수단, 즉 중간 상태다.
- 중간 상태를 파일로 기록하는 과정을 구체화라 한다.
- 유닉스 파이프는 중간 상태를 완전히 구체화하는 대신 작은 인메모리 버퍼만을 사용해 점진적으로 출력을 입력으로 스트리밍한다.
  - 맵리듀스 작업은 입력을 생성하는 모든 선행 작업이 완료됐을 때만 시작 가능하므로 장비들 사이에 부하가 한쪽으로 쏠리거나 변하면 워크플로 전체 수행 시간은 느려진다.
  - 매퍼는 종종 중복되기도 한다.
  - 중간 상태를 저장하는 것은 중간 상태 파일들이 여러 장비에 걸쳐 복제됐다는 의미로 임시 데이터에게는 과잉조치다.
- **데이터플로 엔진**
  - 스파크, 테즈, 플링크 등 분산 일괄 처리 연산을 수행하는 엔진이 개발됐는데 이 엔진들은 전체 워크플로를 독립된 하위 작업으로 나누지 않고 작업 하나로서 다룬다는 점이다.
  - 여러 처리 단계를 통해 데이터 흐름을 명시적으로 모델링하기 때문에 데이터플로 엔진이라 부른다.
  - 입력을 파티셔닝해 병렬화하고 한 함수의 출력을 다른 함수의 입력으로 사용하기 위해 네트워크를 통해 복사한다.
  - 맵리듀스와 달리 유연한 방법으로 함수들을 조합할 수 있는데 이런 함수를 연산자라 부르고 연산자의 출력과 다른 연산자의 입력을 연결하는 여러 가지 선택지를 제공한다.
  - 장점
    - 값비싼 작업인 정렬은 필요할 때만 수행한다.
    - 필요없는 맵 태스크는 없다.
    - 워크플로에 모든 조인과 데이터 의존 관계를 명시적으로 선언하기 때문에 스케줄러가 어느 데이터가 어디에 필요한지에 대한 개요를 가져서 지역성 최적화가 가능하다.
    - 연산자 간 중간 상태는 대개 메모리나 로컬 디스크에 기록하는 것으로 충분한데 HDFS에 중간 상태를 기록할 때보다 I/O가 훨씬 적게 든다. 
    - 연산자들은 입력이 준비되는 즉시 실행을 시작할 수 있다.
    - 새로운 연산자를 실행할 때 이미 존재하는 자바 가상 머신을 재활용할 수 있다.
- **내결함성**
  - 중간 상태를 구체화할 때 생기는 이점은 내구성이다.
  - 중간 상태를 쓰지 않으면 내결함성 확보를 위해 아직 유효한 데이터로부터 계산을 다시 해서 복구한다.
  - 데이터를 재연산할 때 중요한 점은 해당 연산이 결정적인지 아닌지를 파악하는 것인데, 연산자를 재시작해서 재연산된 데이터가 잃어버린 원본 데이터와 같지 않다면 다운스트림 연산자가 이전 데이터와 새로운 데이터 간의 모순을 해결하기 어렵다.
  - 해시 테이블의 원소를 탐색할 때 특정 순서를 보장하지 않거나, 확률 통계 알고리즘이 임의로 생성한 숫자에 명시적으로 의존하는 등 우연히 비결정적 동작이 생기기 쉽다.
  - 중간 데이터가 원천 데이터보다 훨씬 작거나 연산이 CPU 중심적이라면 재연산보다 중간 데이터를 파일로 구체화하는 방식이 더 효과적이다.
- **구체화에 대한 논의**
  - 정렬 연산자는 출력을 생산하기 전에 전체 입력을 소비해야 하므로 일시적이라도 상태를 누적할 필요가 있다.

##### 그래프와 반복 처리
- 추천 엔진 같은 머신러닝이나 랭킹 시스템 분야에서도 그래프 처리의 필요성이 떠오르고 있다.
- 많은 그래프 알고리즘이 한 번에 하나의 간선을 순회하는 방식으로 표현되는데 특정 정보를 전파하기 위해 정점 하나와 인접한 정점을 조인하면서 특정 조건에 도달할 때까지 반복한다. 이런 알고리즘을 이행적 폐쇄라 한다. 
- 맵리듀스는 데이터를 일회성으로 처리하지만 그래프는 완료할 때까지 반복이라는 개념이 있어 이런 알고리즘은 반복적 스타일로 구현된다.
- 맵리듀스는 전체 입력 데이터셋을 읽어서 완전히 새로운 출력 데이터셋을 생상하므로 반복이 비효율적이다.
- **프리글 처리 모델**
  - 일괄 처리 그래프를 최적화하는 방법으로 벌크 동기식 병렬 연산 모델이 널리 사용된다.
  - 정점이 다른 정점으로 메시지를 보내고, 반복할 때마다 개별 정점에서 함수를 호출해 그 정점으로 보내진 모든 메시지를 전달한다.
- **내결함성**
  - 메시지는 일괄 처리가 가능해 통신 중 대기 시간이 발생하지 않아 작업 성능 향상에 도움을 준다.
  - 앞선 박복에서 보낸 모든 메시지는 다음 반복에 도착됨을 보장하기 때문에 다음 반복을 시작하기 전에 앞선 반복은 반드시 끝나야 하고 모든 메시지는 네트워크 상에서 복사돼야 한다.
  - 네트워크 상의 문제로 메시지가 사라지거나 중복되거나 지연되더라도 다음 반복에서 메시지는 목적지 정점에서 정확히 한 번만 처리된다.
  - 내결함성은 반복이 끝나는 시점에 모든 정점의 상태를 주기적으로 체크포인트로 저장함으로써 보장된다.
- **병렬 실행**
  - 정점은 다른 정점으로 메시지를 보낼 때 단순히 정점 ID를 사용해 메시지를 전달한다.
  - 빈번하게 통신하는 정점끼리 같은 장비에 위치하는 식으로 파티셔닝하는게 좋으나 최적화해서 분할하는게 어렵기 때문에 임의로 부여된 정점 ID를 기준으로 단순하게 분할하고 관련성이 높은 정점끼리 그룹화하지 않는다. 때문에 장비 간 통신 오버헤드가 많이 발생한다.
  - 그래프가 단일 컴퓨터 메모리에 넣을 수 있는 크기라면 단일 장비 알고리즘이 분산 일괄 처리보다 훨씬 성능이 좋을 가능성이 높다.

##### 고수준 API와 언어
- 고수준 인터페이스를 사용하면 사용자가 시스템을 생산성 높게 사용할 수 있을 뿐 아니라 장비 수준에서도 작업을 더욱 효율적으로 수행할 수 있다.
- **선언형 질의 언어로 전환**
  - 조인을 수행하는 코드를 작성하는 방식에 비해 관계형 연산자로 조인을 나타내면 프레임워크가 조인 입력의 속성을 분석해 자동으로 최적화해준다.
  - 어떤 조인 알고리즘을 선택하느냐에 따라 일괄 처리 작업의 성능이 크게 달라지고, 선언적인 방법으로 조인을 지정하면 가능하다.
  - 고수준 API에 선언적 측면을 포함하면서 실행 중에 이용할 수 있는 질의 최적화기를 가진다면 일괄 처리 프레임워크는 MPP 데이터와 비슷해지면서 동시에 임의의 코드를 실행하고 임의 형식의 데이터를 읽을 수 있는 확장성을 지녀 유연성은 그대로 유지한다.
- **다양한 분야를 지원하기 위한 전문화**
  - 표준화된 처리 패턴을 재사용 가능하게 구현하는 일은 가치가 있다.
  - 다차원 공간에 주어진 아이템과 근접한 아이템들을 찾는 일종의 유사도 검색 알고리즘인 k 최근접 이웃 알고리즘같은 공간 알고리즘도 유용하다.

### 💡 느낀 점
- 단순하게 불변하는 입력을 어떤 과정을 거쳐 출력으로 만들고, 이 출력을 다른 프로그램의 입력으로 사용하는 유닉스의 철학을 맵리듀스와 데이터플로 엔진에 녹아있는 것을 배웠다. 이는 알고리즘 문제를 해결할 때 작은 단위로 쪼개서 큰 단위의 문제를 해결하는 방식과 유사하단 생각이 들었다.
- 오버헤드를 피하기 위해 중간 단계를 거치지 않는 데이터플로도 정렬이나 데이터 크기의 따라 중간 단계를 구체화 하는 것이 더 나을 수 있다는 것을 보고 상황에 따른 적절한 판단이 중요하다 생각했다. 
- 사실 애플리케이션 단위로만 데이터베이스를 다뤄봐서 훨씬 큰 규모의 작업에 대해 추상적인 개념들만 있었는데, 일괄 처리를 통해 큰 규모에서 배치 작업은 어떤식으로 이뤄지는지 알아갈 수 있었다.
- 적당한 규모의 데이터라면 사실 애플리케이션 배치로 작업을 수행할 수 있을 거 같은데, 규모가 더욱 커지니 애플리케이션 배치로는 분산된 데이터베이스들의 데이터를 감당할 수 없고, 효율적으로 작업하기 위해 일괄 처리를 사용하는 것 같다.
